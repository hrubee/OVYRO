1. Executive Summary

Ovyro is a marketplace for land-only real estate. Landowners (sellers) list parcels with photos, videos, and pricing. Investors (buyers) browse listings, save them to custom lists, and submit negotiation/inquiry forms. Every inquiry becomes a lead delivered to the seller who owns that listing. Sellers can connect their Meta (Facebook) Ads account and Pixel so that Ovyro's listing landing pages fire their pixel and send conversion events back to their ad account — letting each seller run their own Meta ad campaigns pointing at their Ovyro listing page.

Three surfaces:

  1. Admin (Platform Owner) view — full business analytics: buyer counts, seller counts, Meta-connected sellers, listings, leads, funnel health, moderation tools.
  2. Buyer (Consumer) view — browse/search listings, inquire/negotiate, profile, saved custom lists ("like" button), history of inquiries, and an opt-in path to become a seller.
  3. Seller view — everything a buyer can do, plus listing management (photos/videos/details), a lead inbox, and Meta Ads/Pixel connection.

Role hierarchy rule (explicit requirement): Seller ⊇ Buyer. A user who signs up as a seller automatically has the full buyer experience. A buyer does not get seller capabilities by default — they must complete a seller onboarding flow to upgrade.

2. Assumptions & Open Questions

These are decisions the spec assumes. Each is safe to build with, but flag them before launch.

# Assumption Why it matters Recommended default
A1 No payments/escrow on-platform in v1. Leads only. Removes licensing, escrow, and compliance burden at launch. Confirm; add payments in a later phase if ever.
A2 Single market/country at launch; single currency. Currency, units (acres vs. hectares vs. sq ft), and legal disclosures vary by market. Make currency & area unit configurable per listing but default to one.
A3 Sellers are individuals or small firms, not enterprise brokerages (no MLS integration in v1). MLS/IDX integration is a large separate project. Skip MLS in v1.
A4 Listing landing pages are public (no login required to view). Required for Meta ads to work — ad traffic must land on a public page. Public listing pages; login required only to save lists or submit an inquiry (see L-2 note in §5.2 for a friction tradeoff).
A5 "Negotiation" in v1 = structured inquiry form (offer amount + message), not real-time chat. Chat is a large scope increase; lead delivery is the core loop. Structured offers first; threaded messaging in Phase 2.
A6 Admin = a small internal team, not a self-serve multi-tenant admin. Simplifies RBAC. One admin role, optionally admin:read_only later.
A7 Meta integration scope = Pixel firing + Conversions API (CAPI) events + "is connected" status. Ovyro does not create or manage ad campaigns on the seller's behalf in v1. Full campaign management via Marketing API is a major scope increase and requires advanced Meta App Review. Pixel + CAPI + connection status in v1; campaign insights read-only in Phase 2.

Open questions to answer before build starts:

  1. Which country/market first? (Drives legal disclaimers, phone number formats, map provider choice, area units.)
  2. Is seller listing free, or is there a listing fee / subscription / per-lead fee? (Monetization affects onboarding and admin analytics; spec assumes free listings in v1 with monetization hooks left in the schema.)
  3. Do leads include the buyer's contact info immediately, or only after seller accepts? (Spec assumes immediate delivery — simplest and matches "leads are sent to the offer creator.")
  4. Should buyers see the seller's identity/contact, or is all contact routed through the platform? (Spec assumes seller display name + platform-routed contact to reduce disintermediation.)

3. Roles & Permission Model

3.1 Roles

  • buyer — default role for every registered user.
  • seller — additive role granted after completing seller onboarding. Always retains buyer.
  • admin — platform owner/staff. Assigned manually (seed script / direct DB or admin invite), never self-serve.

Implementation: a roles array (or join table) on the user, not a single mutually-exclusive role column. This is important — the "seller has everything a buyer has" requirement breaks if role is a single enum and someone writes if role === 'seller' checks. All buyer-facing features must check "is authenticated," never "role === buyer."

3.2 Permission matrix

Capability Anonymous Buyer Seller Admin
Browse/search listings, view listing pages ✅ ✅ ✅ ✅
Submit inquiry/negotiation on a listing ⛔ (prompted to sign up) ✅ ✅ (not on own listings) ⛔
Create custom lists / like listings ⛔ ✅ ✅ —
View own inquiry history ⛔ ✅ ✅ —
Start seller onboarding ⛔ ✅ n/a —
Create/edit/pause own listings ⛔ ⛔ ✅ ✅ (any, for moderation)
View lead inbox for own listings ⛔ ⛔ ✅ ✅ (all leads, read-only)
Connect Meta ad account / pixel ⛔ ⛔ ✅ —

Guard rails:

  • A seller cannot inquire on their own listing (prevents fake-lead inflation of their own stats).
  • Ownership checks on every listing/lead/media mutation: listing.seller_id === session.user.id OR isAdmin.
  • Admin actions are audit-logged (who, what, when, before/after snapshot).

4. Feature Specification by View

4.1 Admin View (Platform Owner)

Route namespace: /admin/*, gated by admin role, hidden from all navigation for non-admins.

4.1.1 Overview dashboard (/admin) Headline KPI cards, each with a trend vs. previous period (7/30/90-day selector):

  • Total registered users; split: buyers-only vs. sellers.
  • New signups in period (buyers, sellers).
  • Active listings / paused / pending-review / rejected counts.
  • Sellers with Meta connected — count and % of all sellers; broken into "ad account connected," "pixel configured," and "both + events flowing in last 7 days" (a connection that sends no events is effectively broken — surface that distinctly).
  • Total leads in period; leads per listing (median); lead → seller-response rate (Phase 2, once lead status tracking is live).
  • Funnel: listing page views → inquiry form starts → inquiries submitted (period, with conversion %).

4.1.2 Users (/admin/users) Searchable/filterable table: name, email, roles, signup date, # inquiries made, # listings (if seller), Meta connection status, last active. Row actions: view detail, suspend/unsuspend, grant/revoke seller role (manual override for onboarding edge cases), delete (soft-delete with GDPR-style anonymization).

4.1.3 Listings (/admin/listings) Moderation queue is the primary job: new listings land in pending_review and require admin approval before going live (land fraud is common; see §11 R-1). Filters: status, seller, date, flag reports. Actions: approve, reject with reason (emailed to seller), unpublish, feature/unfeature, edit.

4.1.4 Leads (/admin/leads) Read-only view of all leads across the platform: listing, buyer, seller, offer amount, timestamp, delivery status (delivered via email? seller viewed it?). Purpose: dispute resolution and funnel debugging, not lead management.

4.1.5 Analytics (/admin/analytics) Deeper charts (time series): signups, listings created, leads, page views by listing, top listings by leads, geographic distribution of listings and of buyer interest, seller Meta-connection adoption over time. Implementation detail in §10.

4.1.6 Settings (/admin/settings) Feature flags, moderation toggles (auto-approve on/off), email template previews, admin user management.

4.2 Buyer (Consumer) View

4.2.1 Public browse & search (/, /land, /land/[slug])

  • Grid/list of active listings with photo, title, price, area, location, land type (agricultural / residential plot / commercial / recreational / other).
  • Filters: location (text + region select; map-radius search in Phase 2), price range, area range (with unit), land type, road access, utilities available, sort (newest, price, area, popularity).
  • Search: Postgres full-text on title/description/location in v1. (PostGIS geo-radius is Phase 2 — schema includes lat/lng from day one so no migration pain.)
  • Listing detail page = the seller landing page (see §5.2): gallery (photos + videos), full description, parcel details (area, zoning if provided, access, utilities, coordinates/map embed), price & negotiability flag, seller display name, inquiry form, like/save button, share button. This page must be fast, public, SEO-friendly (SSR, OpenGraph tags) — it's the destination for sellers' Meta ads.

4.2.2 Inquiry / negotiation form On each listing: "Contact seller / Make an offer." Fields: name & phone (pre-filled from profile), offer amount (optional — "ask at listed price" default), message, preferred contact method, consent checkbox (privacy policy + "share my contact details with the seller"). Behavior:

  • Requires login; anonymous users hit a low-friction signup wall (email or phone OTP) with the form state preserved.
  • On submit: lead record created → seller notified (email immediately; in-app inbox; SMS/WhatsApp Phase 2) → buyer gets confirmation → Meta conversion event Lead fired for that listing's seller pixel (pixel client-side + CAPI server-side with shared event_id, §5.3).
  • Anti-abuse: rate limit (e.g., max 10 inquiries/day/user, 1 per listing per 72h), honeypot field, optional Turnstile/hCaptcha, phone verification via OTP before first inquiry (strongly recommended — sellers churn instantly on junk leads; see §11 R-3).

4.2.3 Profile & account (/account)

  • Auth: email+password and email OTP magic link; phone OTP for verification; Google OAuth optional. Sessions via secure HTTP-only cookies.
  • Profile: name, phone (verified badge), email, avatar, investment preferences (regions, budget range, land types) — used later for alerts/recommendations.
  • My Inquiries (/account/inquiries): every listing the buyer has inquired on, with their offer, date, and listing status (still active / sold / removed). This is the required "see all listings I've requested to negotiate on."
  • My Lists (/account/lists): the like/wishlist system:
    • Every user gets a default "Saved" list.
    • Users can create unlimited custom-named lists ("Goa beach plots," "Under 50L near highway," etc.), rename, delete.
    • Heart/save button on every listing card and detail page → quick-save to "Saved" or long-press/dropdown to pick/create a list. A listing can be in multiple lists.
    • Lists show current price and a "price changed since you saved" indicator (nice-to-have, cheap to build: store price_at_save).
    • If a saved listing is sold/removed, it stays in the list greyed-out with status badge (don't silently disappear — investors track outcomes).
  • Become a Seller CTA: persistent but unobtrusive entry point in account menu and after a buyer's Nth inquiry ("Own land? List it on Ovyro").

4.2.4 Buyer → Seller onboarding (/sell/onboarding) Multi-step, resumable:

  1. Intent & type (individual owner / broker / company).
  2. Contact & identity: legal name, verified phone (OTP), address; optionally an ID document upload (feature-flagged — enable when fraud appears; see §11 R-1).
  3. Payout/monetization details — skipped in v1 (no payments), slot reserved.
  4. Agreement: seller terms acceptance (timestamped).
  5. Completion → seller role added, redirected to seller dashboard with a "create your first listing" wizard.

State machine: not_started → in_progress(step_n) → submitted → approved | rejected. If admin review of sellers is enabled (flag), submitted requires approval; otherwise auto-approve.

4.3 Seller View

Route namespace: /dashboard/* (seller-gated). Global nav shows a role switcher ("Browsing" / "Selling") so the seller ⊇ buyer requirement is a first-class UI concept, not two separate accounts.

4.3.1 Listings management (/dashboard/listings)

  • Create/edit wizard: basics (title, land type, price, negotiable flag) → location (address text, map-pin drag for lat/lng, plot/survey number optional) → details (area + unit, dimensions, road access, utilities, zoning, legal docs available y/n, description) → media → review & submit.
  • Media: up to 25 photos + 5 videos per listing. Direct-to-storage uploads via presigned URLs (never through the app server — Railway containers shouldn't buffer 500MB videos). Photos: jpg/png/webp/heic, ≤15MB each, server-side processed to responsive webp variants + blur placeholder. Videos: mp4/mov ≤500MB, transcoded to HLS via Mux (recommended) or Cloudflare Stream; show processing status. Drag-to-reorder; first photo = cover.
  • Statuses: draft → pending_review → active → paused | sold | rejected | expired. Seller can pause/reactivate and mark sold. Listings auto-expire after 90 days with renew prompts (keeps inventory fresh; configurable).
  • Per-listing stats: views, saves, inquiries (7/30-day sparkline).

4.3.2 Lead inbox (/dashboard/leads)

  • All inquiries across the seller's listings; filter by listing, date, status.
  • Lead card: buyer name, verified-phone badge, offer amount, message, contact details, timestamps.
  • Status pipeline seller can set: new → contacted → negotiating → won → lost (feeds admin funnel analytics and gives sellers a lightweight CRM).
  • Every lead also delivered by email at submission time (leads are the product — never rely on the seller logging in).

4.3.3 Meta Ads connection (/dashboard/marketing) Full detail in §5. Seller-facing surface:

  • "Connect Meta" button → Facebook OAuth (Facebook Login for Business).
  • After connect: pick ad account, pick (or auto-detect) Pixel/dataset ID.
  • Status panel: token health (valid / expiring / expired — reconnect CTA), pixel ID in use, events sent in last 7 days per listing, test-event tool ("fire a test Lead event and check Events Manager").
  • Clear explanation copy: "Your Pixel fires on your listing pages. Run Meta ads pointing to your listing URL, and lead conversions flow into your ad account for optimization."
  • Disconnect button (deletes tokens, stops pixel injection).

4.3.4 Seller settings Business profile (display name shown to buyers, about, logo), notification preferences (email per-lead, daily digest), and everything from buyer account settings.

5. Meta Ads / Pixel Integration (Deep Dive)

This is the most technically sensitive part of the platform. Requirements restated precisely:

  • All seller listing pages share the same template on Ovyro's domain.
  • Each seller can connect their own Meta ad account and Pixel.
  • When a buyer converts (submits an inquiry) on Seller X's listing page, the conversion must be reported to Seller X's pixel/ad account — and no one else's.

5.1 Meta App setup (one-time, platform-level)

  • Create a Meta App (Business type) owned by Ovyro's Meta Business Manager.
  • Products: Facebook Login for Business + Marketing API (for CAPI + pixel listing).
  • Permissions to request in App Review: ads_management (or ads_read if only reading), business_management, pages_show_list (only if needed). Start development in Dev Mode with test users; App Review + Business Verification are hard prerequisites for real sellers — begin this process at project start, it takes weeks (see §11 R-6).
  • Use long-lived tokens (60-day) exchanged from the OAuth short-lived token. Store: encrypted at rest (AES-256-GCM via app-level encryption; key in Railway env var), never sent to the client, never logged.

5.2 Pixel injection on listing pages

  • meta_connections table maps seller → pixel_id (+ token, ad_account_id).
  • Listing page SSR: if the listing's seller has an active connection with a pixel_id, inject the Meta Pixel base code with that pixel_id only, and fire PageView + ViewContent (content_ids: [listing_id], content_type: 'product', value: price, currency).
  • Isolation rule (critical): exactly one seller pixel per page — the listing owner's. Never fire Seller A's pixel on Seller B's page. Ovyro's own platform pixel (if any) may fire alongside, but seller pixels never cross listings. Enforce in one single code path (a <MetaPixel sellerConnection={…}/> component) so there is exactly one place this can go wrong.
  • L-2 friction note: ads optimized for Lead events perform poorly if a login wall blocks the form. Mitigate: the inquiry form itself is fillable without login; the OTP verification happens inline as the final step of submission (verify phone → lead created). This keeps ad conversion friction low while still verifying every lead.
  • Consent (GDPR/ePrivacy): if operating anywhere consent laws apply, pixels load only after cookie-consent acceptance. Build the consent banner + conditional pixel loading from day one — retrofitting is painful. In consent-denied cases, only server-side CAPI with limited data (or nothing) fires, per policy configuration.

5.3 Conversions API (server-side) + deduplication

Client-side pixels are increasingly blocked (ad blockers, iOS). Send every conversion both ways:

  1. Browser: fbq('track', 'Lead', {value, currency, content_ids:[listing_id]}, {eventID}).
  2. Server (background job, not in the request path): POST to https://graph.facebook.com/v{N}/{pixel_id}/events with the seller's token, same event_id, event_name: 'Lead', event_source_url, hashed user data (em, ph — SHA-256 of normalized email/phone), fbp/fbc cookies captured at submit, action_source: 'website'.
  3. Meta dedupes on event_id + event name. Generate event_id server-side at lead creation (ULID) and pass to the client.

Events to implement: PageView, ViewContent (listing view), Lead (inquiry submitted). Optional Phase 2: AddToWishlist (save button), InitiateCheckout-analog (form started).

CAPI dispatch runs on a queue with retries + exponential backoff; failures visible in seller status panel and admin analytics ("events flowing" metric, §4.1.1).

5.4 Token lifecycle & failure modes

Failure Detection Handling
Token expired (60d) or revoked by seller on Meta side CAPI call returns 190/463 error codes; scheduled daily token-check job Mark connection needs_reauth; stop pixel injection? No — pixel injection needs no token (pixel_id is public-ish), so keep firing pixel; only CAPI pauses. Email seller + dashboard banner: "Reconnect Meta."
Seller deletes pixel in Meta CAPI errors reference invalid pixel Same as above; prompt reconfigure.
Seller connects the wrong ad account/pixel Confusing zero-event debugging Test-event tool in dashboard (§4.3.3) + show pixel name fetched from API, not just ID.
Meta rate limits 429 / error subcodes Queue backoff; batch events (CAPI accepts arrays).
App Review not granted at launch Blocking Feature-flag the whole Meta module; platform launches without it, sellers see "coming soon."

5.5 What Ovyro explicitly does NOT do in v1

  • Create/manage campaigns, ad sets, or ads for sellers.
  • Spend money or hold ad budgets.
  • Show campaign performance dashboards (Phase 2: read-only insights via ads_read). This keeps App Review scope minimal and avoids becoming an ad agency by accident.

6. Data Model (PostgreSQL)

Conventions: ULID primary keys (id text primary key), created_at/updated_at timestamptz on all tables, soft-delete via deleted_at where noted. Prisma or Drizzle ORM; migrations checked into repo.

8. Tech Stack & Architecture

Chosen for: single-repo simplicity, first-class Railway support, and being a stack Claude Code builds extremely well.

Layer Choice Rationale
Framework Next.js 15 (App Router, TypeScript) — one app serves public pages (SSR/ISR for SEO-critical listing pages), buyer/seller dashboards, admin, and API routes One deployable service; listing pages need SSR for SEO + OG tags + per-seller pixel injection
DB PostgreSQL (Railway plugin) + Drizzle ORM (or Prisma) Relational fits marketplace data; Railway-native
Cache/queue Redis (Railway plugin) + BullMQ CAPI dispatch, emails, transcode callbacks, token checks, listing expiry — all queued
Worker Second Railway service, same repo/image, start command node worker.js Never run queues inside the web process — Railway restarts/deploys would drop jobs
Object storage Cloudflare R2 (S3-compatible) Railway has no object storage; R2 has zero egress fees (media-heavy platform)
Video Mux (or Cloudflare Stream) Transcoding, HLS, thumbnails, webhooks — never DIY ffmpeg in v1
Images sharp in worker → webp variants + blurhash; serve via R2 + Cloudflare CDN Cheap, standard
Auth Better Auth (or Auth.js) with email/password, email OTP, phone OTP (Twilio Verify / MSG91 depending on market), Google OAuth Self-hosted = no per-MAU cost; phone OTP is required for lead quality anyway
Email Resend or Postmark, React Email templates Lead notifications are business-critical: use a transactional-grade provider, set up SPF/DKIM/DMARC day one
Maps Mapbox GL (or Google Maps if market dictates) Map pin picker + listing map embed
Validation Zod everywhere (API + forms via react-hook-form)
UI Tailwind + shadcn/ui Fast, consistent, Claude-Code-friendly
Monitoring Sentry (errors) + Railway logs; /api/health endpoint
Rate limiting Redis sliding window (per-IP + per-user) on auth, leads, media presign

8.1 Railway deployment topology

Services in one Railway project:

  1. web — Next.js (next start), healthcheck /api/health, min 1 replica (scale horizontally later; app must stay stateless — sessions in DB/Redis, uploads direct-to-R2).
  2. worker — BullMQ consumers (emails, CAPI, media, token-check, expiry).
  3. cron — Railway cron triggers hitting worker jobs (daily token health check, listing expiry sweep, analytics rollups) — or repeatable BullMQ jobs inside worker (simpler; pick this).
  4. Postgres plugin, Redis plugin.

Deploy pipeline:

11. Risks, Edge Cases & Mitigations

R-1. Fraudulent / disputed land listings (highest business risk). Land fraud (fake ownership, disputed titles, agricultural land sold as residential) is endemic. Mitigations: admin approval queue before any listing goes live (v1 default ON); "legal docs available" self-declaration surfaced on listing; report-listing button; prominent disclaimer on every listing + lead email ("Ovyro is a discovery platform; verify title independently before any payment"); seller identity phone-verified, ID-doc upload behind a feature flag ready to enable; repeat-offender suspension tooling in admin.

R-2. Disintermediation. Buyers and sellers will transact off-platform — in a lead-gen model this is fine (the lead is the product). Do not build contact-hiding friction that degrades lead quality. Revisit only if monetization becomes per-transaction.

R-3. Junk/spam leads kill seller trust. Mitigations: phone OTP before first inquiry (mandatory), per-user and per-IP rate limits, honeypot + Turnstile, dedupe (one inquiry per listing per buyer per 72h), admin visibility into lead volumes per buyer, ban tooling. Seller-side "report spam lead" in Phase 2.

R-4. Seller-pixel cross-contamination. Firing the wrong seller's pixel = data leakage between competitors and a trust catastrophe. Mitigations: single code path for pixel injection (§5.2), unit test asserting listing page HTML contains exactly the owner's pixel_id and no other connection's, e2e test with two seeded sellers.

R-5. Meta token/permission fragility. Covered in §5.4 — token check job, needs_reauth state, pixel keeps firing without token, CAPI queue with retries, seller status panel.

R-6. Meta App Review / Business Verification delays (weeks–months). Mitigations: start verification at project kickoff; whole Meta module behind a feature flag; dev mode works for internal test sellers meanwhile; launch does not block on it.

R-7. Large media uploads on Railway. Containers have limited disk and request-size tolerance. Mitigation: presigned direct-to-R2 uploads only; worker downloads for processing to a temp dir with size guards; Mux handles video entirely (pull from R2 URL).

R-8. SEO + ad landing page performance. Meta ad costs rise with slow pages. Mitigations: SSR + edge-cached listing pages (ISR revalidate on listing update), responsive webp images, LCP budget < 2.5s on 4G, OG image auto-generated per listing.

R-9. Duplicate accounts / seller inquiring on own listing via second account. Self-inquiry blocked by user id; duplicate detection via phone number uniqueness (one verified phone → one account).

R-10. GDPR/DPDP-style privacy obligations. Consent checkbox on lead form (stored with timestamp), cookie consent gating pixels, account deletion flow (anonymize user, retain leads with contact redacted after retention window), Meta-required data-deletion callback implemented, privacy policy + seller terms pages as real routes.

R-11. Buyer→seller upgrade edge cases. User mid-onboarding must keep full buyer functionality; abandoning onboarding is resumable; rejection doesn't affect buyer role; a suspended seller keeps buyer access unless account-level suspended.

R-12. Data integrity for "sold" listings. Sold/removed listings stay resolvable (greyed-out state) for lists and inquiry history — never hard-delete a listing with leads; use status + soft delete.

12. Non-Functional Requirements

  • Security: OWASP basics; all mutations CSRF-protected (same-site cookies + framework protections); RBAC middleware on /dashboard, /admin, and every API route; encrypted Meta tokens; secrets only in Railway env; dependency audit in CI.
  • Performance: listing browse P95 < 500ms server time; listing page LCP < 2.5s; lead submit P95 < 800ms (Meta CAPI is async, never inline).
  • Availability: single-region acceptable in v1; healthchecks + Railway restart policy; queue-backed side effects mean a worker outage delays (not loses) emails/CAPI.
  • Testing: unit (lib logic, pixel-isolation test R-4), integration (API routes against test DB), one happy-path e2e per role (Playwright): buyer inquiry flow, seller listing+lead flow, admin approve flow.
  • Accessibility: semantic forms, keyboard-navigable galleries, alt text prompts on media upload.
  • i18n-readiness: copy in a messages file even if single-language at launch; currency/area unit formatting centralized.
